## 一、整体架构概述

### 1.1 目标

灵基产品采用「基础订阅 + AI点数」的极简定价模式。计量计费系统的目标：

1. **成本可控**：客户清楚知道"用了多少、花了多少、还能用多久"
2. **计费准确**：所有第三方服务成本能被点数收入覆盖
3. **可追溯**：计量记录不可修改、不可删除，支持审计对账
4. **可扩展**：支持新服务、新模型的快速接入

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| 计量与计费解耦 | 计量记录原始消耗事实，计费由引擎计算点数 |
| 统一计费单元 | 全局使用唯一单位 Unit（点数） |
| 配置化驱动 | 计费规则存数据库，动态解析，不硬编码 |
| 先检后用 | 调用前检查余额，余额不足则熔断 |
| 流水不可逆 | 余额每次变化都记录流水，永久保存 |

### 1.3 要做的内容

| 阶段 | 内容 | 计费方式 |
|------|------|---------|
| ✅ 已完成 | 大模型（LLM）计量计费 | Token × 系数 ÷ 10000 |
| 🔜 马上做 | WebSearch、OCR 等非LLM服务 | 次数 × 费率公式 |
| 📋 后续做 | 预警、账单、统计分析、统一网关等 | - |

### 1.4 涉及系统及职责

| 系统 | 角色 | 职责 |
|------|------|------|
| **kdmanage-backend** | 中心管控 | 系数管理、版本化、许可导入、额度推送、跨环境调度 |
| **kdmanage-frontend** | 管控前端 | 系数维护页面、许可管理、配额比率管理 |
| **tmanage-backend** | 租户计量引擎 | 从MAAS拉取消耗、Token→点数计算、扣点、流水、缓存、仪表板数据 |
| **tmanage-frontend** | 租户前端 | 计量仪表板、成员配额管理、沙箱分配 |
| **MAAS** | 大模型网关 | 路由调用、记录Token消耗、推送用量明细 |
| **Console/计量服务** | 计量接口 | 非LLM服务的统一计量入口（带鉴权转发） |

### 1.5 数据流总览

```
kdmanage（管控）
  │  推送系数、额度、许可
  ↓
tmanage（租户计量引擎）
  │  ① 定时从MAAS拉取LLM消耗
  │  ② 接收非LLM计量接口上报
  │  ③ Token/次数 × 系数 = 点数
  │  ④ 扣减配额（先席位→后共享池）
  │  ⑤ 记录流水 + 刷新缓存
  ↓
tmanage-frontend（仪表板）
  │  展示消费趋势、排名、配额使用率
```

## 二、已实现功能

### 2.1 计量（Metering）

**数据来源**：定时任务从MAAS拉取Token消耗明细

**核心实现**：`MeteringDeductionServiceImpl.runOnce()`

**流程**：
1. 刷新系数缓存（SysRatioCache.reload）
2. 读取拉取水位线游标（PullCursor）
3. 限制窗口最长5分钟
4. 从MAAS拉取用量明细（WebClientMaasUsageClient）
5. 逐条计算点数 + 写入计量表

**计量记录表**：`t_meter_tenant_user_llm_metering`

| 字段 | 说明 |
|------|------|
| tenant_id, user_id | 谁调的 |
| input_tokens, output_tokens, cached_tokens | 消耗多少Token |
| llm_id, llm_version | 调的什么模型 |
| value | 计算出的点数 |
| ratio_version | 用的哪个版本的系数 |
| request_id | MAAS请求ID（幂等） |
| agent_id, skill_id | 调用链（智能体、技能） |
| create_time | 调用时间 |

### 2.2 Token → 点数计算

**核心方法**：`MeteringDeductionServiceImpl.getBigDecimal()`

**当前公式**：
```
points = (inputTokens - cachedTokens) × inputRatio
       + outputTokens × outputRatio
       + cachedTokens × cacheInputRatio
       ÷ conversionRate（默认10000）
       向上取整到4位小数
```

**升级后公式**（详见[《大模型Token计费规则配置升级方案》](./【设计】t_meter_sys_ratio表结构升级方案.md)）：
```
points = (
    文本输入Token × 文本输入系数
  + 图片输入Token × (图片输入系数 ?? 文本输入系数)
  + 视频输入Token × (视频输入系数 ?? 文本输入系数)
  + 音频输入Token × (音频输入系数 ?? 文本输入系数)
  + 文本缓存命中Token × (文本缓存命中系数 ?? 文本输入系数)
  + 音频缓存命中Token × (音频缓存命中系数 ?? 文本缓存命中系数)
  + 缓存写入Token × (缓存写入系数 ?? 文本输入系数)
  + 文本输出Token × 文本输出系数
  + 图片输出Token × (图片输出系数 ?? 文本输出系数)
  + 视频输出Token × (视频输出系数 ?? 文本输出系数)
  + 音频输出Token × (音频输出系数 ?? 文本输出系数)
  + 推理Token × (推理输出系数 ?? 文本输出系数)
) ÷ 换算基数
```

新增覆盖维度：多模态分价（文本/图片/视频/音频）、缓存写入、推理Token、上下文分段、峰谷时段。

**换算基数**：国内模型基准价5元对应100万tokens，国外模型基准价$1对应100万tokens。

**系数来源**：
- kdmanage维护 → 推送到tmanage → `t_meter_sys_ratio` → `SysRatioCache`内存缓存
- 找不到系数时用配置默认值兜底

### 2.3 扣点（Deduction）

**核心方法**：`DeductionExecutor.deductOne()`

**扣减顺序**：
```
① 先扣席位配额（item_type=0，跟人走，amount-meter=可用）
② 累加到group表（统计用，不限额）
③ 累加到user表（share_meter，统计用）
④ 循环扣共享池（item_type=1，按deduction_order排序，校验有效期）
⑤ 所有池耗尽 → 生成OVERDRAFT透支流水
```

**一次调用可能产生多条流水**（如席位不够用再扣共享池）。

### 2.4 流水（Statement）

**表**：`t_meter_tenant_user_statement`（按月分表）

**流水类型**：

| 类型 | value方向 | 说明 |
|------|----------|------|
| SEAT | 负值 | 席位配额扣减 |
| CREDIT | 负值 | 共享池扣减 |
| OVERDRAFT | 负值 | 透支（所有池耗尽） |
| SEAT_GRANT | 正值 | 席位额度发放 |
| CREDIT_ADD | 正值 | 共享池额度发放 |
| CREDIT_GRANT | 正值 | 赠送点数 |
| CREDIT_EXPIRE | 负值 | 点数过期清零 |

### 2.5 配额检查（调用前熔断）

**服务**：`QuotaCheckService`

**检查流程**：
1. 查租户级缓存 `quota:{tenantId}`
2. 查用户级缓存 `quota:{tenantId}:{userId}`
3. 任一有余量（meter < amount）→ 放行
4. 均无余量 → 拒绝调用

### 2.6 配套定时任务

| 任务 | 周期 | 功能 |
|------|------|------|
| MeteringDeductionTask | 高频（秒级） | 从MAAS拉取并扣减 |
| SeatQuotaResetTask | 月末 | 席位配额清零 |
| QuotaExpirySettlementJob | 日检 | 点数包过期结算 |

### 2.7 前端功能

**kdmanage-frontend**：
- 大模型系数管理（QuotaRatiosPage）：CRUD + 版本化
- 许可管理（LicensesPage）：查看、详情
- 许可包/SKU/特性配置

**tmanage-frontend**：
- 计量仪表板（MeteringDashboardPage）：
  - 席位使用率 + 资源池使用率 + 今日消耗
  - 成员配额表（分配、编辑、移除、批量操作）
  - 沙箱额度分配（仅生产环境）
  - 套餐详情查看

### 2.8 跨环境管理

- ✅ 生产 ↔ 沙箱 额度转移（`createTransferLicense`、`SandboxTmanageClient`）
- ✅ 许可按环境区分（production/sandbox）

## 三、马上要做：多模式计费（非LLM服务）

### 3.1 概述

当前计量只支持LLM（大模型Token）。接下来要扩展到所有非LLM服务，统一使用点数计费。

**优先级**：WebSearch → OCR → 其他（语音、图像、WPS等）

### 3.2 接入方式：统一委托授权模式

**核心原则**：所有第三方服务调用都通过统一代理，业务方不直接调用也不自行上报，保证每次调用都被可信记录。

**按代理方不同分两种路径**：

| 路径 | 代理方 | 适用场景 | 原因 |
|------|--------|---------|------|
| **MAAS代理** | MAAS网关 | WebSearch等与LLM调用耦合的服务 | 搜索请求随大模型调用一起发出，由MAAS统一管理 |
| **Console/计量服务代理** | Console计量接口 | OCR、图像生成、语音等独立服务 | 与LLM无关，由计量服务统一转发和记录 |

#### 路径A：MAAS代理（WebSearch）

```
harness/build → 走MAAS进行WebSearch调用
    → MAAS调用外部LLM + 外接WebSearch服务
    → MAAS记录websearch调用次数
    → MAAS异步推送（LLM token + websearch次数）
    → Console接收 → 按费率转换为点数 → 扣点
```

**说明**：
- LLM内置联网搜索（厂商自带能力）：费用已含在Token定价中，走现有LLM Token计量
- 外接WebSearch（独立付费服务）：由MAAS记录次数，推送到Console独立计算点数

> ⚠️ **待确认**：外接WebSearch服务商的收费方式（按次/按量/其他），以便配置对应的计费公式。

#### 路径B：Console/计量服务代理（OCR等独立服务）

```
harness/build → 调用Console的计量接口（带租户用户鉴权）
    → Console转发到外部服务（OCR等）
    → 外部服务执行功能
    → Console记录调用并按公式转点数
    → Console异步扣点
```

**为什么用委托授权**：
- 计量服务统一代理调用，保证每次调用都被记录
- 避免业务方"瞒报/漏报"
- 鉴权由计量接口统一管理，外部服务不需要知道租户/用户信息

### 3.4 技术实现（来自开发spec）

#### 新增统一计量事件表

**表名**：`t_meter_tenant_user_other_metering`

（字段详见[《非LLM计费公式配置方案》](./【设计】非LLM计费公式配置方案.md)）

#### 新增计费公式表

**表名**：`t_meter_sys_metering_formula`

**唯一标识**：计费类型 + 供应商 + 服务规格

**6种预设模板**：

| 模板 | 名称 | 适用场景 | 公式 |
|------|------|---------|------|
| T1 | 按次计费 | WebSearch、WPS | `unit_price × metric_volume` |
| T2 | 按时长计费 | 语音识别、视频 | `unit_price × Math.ceil(duration / billing_unit)` |
| T3 | 按字符量计费 | TTS | `unit_price × Math.ceil(char_count / billing_unit)` |
| T4 | 按数量计费 | 图像生成 | `unit_price × metric_volume` |
| T5 | 阶梯计费 | OCR大量、ASR大量 | 按月累计量分段取单价 |
| T6 | 自定义公式 | 复杂场景（TODO） | QLExpress表达式 |

**阶梯支持两种模式**：超额计费（电费模式）和 全量计费（档位模式）

详见[《非LLM计费公式配置方案》](./【设计】非LLM计费公式配置方案.md)

#### 计费引擎

- **技术选型**：QLExpress（阿里系表达式引擎）
- **关键要求**：全程BigDecimal + 公式预编译缓存

#### 流水表

复用现有 `t_meter_tenant_user_statement`，通过 `metering_type` 字段区分LLM和非LLM。

### 3.5 配套需求（详见第五段）

- 公式管理界面（费率配置）
- 非LLM消费的统计展示

## 四、后续要做

### 4.1 预警机制

**设计**（已定义，前端未完善）：

| 阶段 | 使用率 | 行为 |
|------|--------|------|
| 正常 | < 80% | 无限制 |
| 黄色预警 | 80% ~ 100% | 无限制 + 发送预警通知 |
| 豁免区 | 100% ~ 120% | 无限制 + 不额外计费 |
| 红色熔断 | > 120% | 拒绝所有调用 |

**待做**：前端预警展示、通知方式（站内信/短信/邮件）

### 4.2 账单生成

- 按月出账单
