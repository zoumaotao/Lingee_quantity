# 【设计】非LLM计费公式配置方案

> **版本**：v1.1  
> **编制人**：邹茂桃  
> **时间**：2026-07-03  
> **目标**：覆盖WebSearch、OCR、图像生成、语音识别等非LLM服务的计费配置

---

## 一、为什么要做

### 1.1 和LLM的核心区别

| 维度 | LLM计费 | 非LLM计费 |
|------|--------|-----------|
| 计量原子 | 固定（Token） | **不固定**（次数/秒数/张数/字符数…） |
| 公式结构 | 固定（Token×系数÷基数） | **不固定**（按次/按时长取整/阶梯/自定义） |
| 配置什么 | 只配系数值 | **既配公式结构，又配参数值** |
| 计量来源 | MAAS定时推送 | Console计量接口实时上报（委托授权） |

### 1.2 接入优先级

1. **WebSearch** — 按次，最简单
2. **OCR** — 按次，有阶梯
3. **图像生成** — 按张，按规格
4. **语音识别（ASR）** — 按时长取整
5. **TTS文字转语音** — 按字符数
6. **WPS** — 按次/按页
7. **视频生成** — TODO（当前无业务场景，由自定义公式兜底扩展）

---

## 二、主流服务厂商计价模式

| 服务类型 | 计量单位 | 计价特点 | 是否阶梯 | 是否峰谷 |
|---------|---------|---------|---------|---------|
| WebSearch | 次 | 统一单价 | ❌ | ❌ |
| OCR文字识别 | 次 | 按次，按子类型不同价 | ✅ 月累计阶梯 | ❌ |
| 语音识别(ASR) | 15秒（计费单位） | 按时长向上取整到计费单位 | ✅ 月累计阶梯 | ❌ |
| 图像生成 | 张 | 按张，按分辨率分档 | ❌ | ❌ |
| 文字转语音(TTS) | 千字符 | 按字符数向上取整 | ✅ 月累计阶梯 | ❌ |
| WPS文档处理 | 次/页 | 统一单价 | 可能 | ❌ |
| 视频生成 | 秒 | 按时长×分辨率 | ❌ | ❌ |

**结论**：
- 阶梯计价常见（OCR、ASR、TTS）
- 峰谷在非LLM服务中不存在
- 同一服务内按子类型/规格不同定价（如OCR通用 vs 发票，图像1K vs 4K）

## 三、模板设计

### 3.1 设计原则

- **保留细分模板名称**：配置人员看到"按时长计费"比看到"T2"更直观
- **底层公式可复用**：部分模板公式结构相似，但展示和配置引导不同
- **兜底自定义**：覆盖未来未知场景

### 3.2 模板清单

| 模板 | 名称 | 适用场景 | 公式 | 配置人员需要填什么 |
|------|------|---------|------|------------------|
| **T1** | 按次计费 | WebSearch、WPS | `unit_price × metric_volume` | 单价 |
| **T2** | 按时长计费 | 语音识别 | `unit_price × Math.ceil(duration_seconds / billing_unit_seconds)` | 单价、计费单位（如15秒） |
| **T3** | 按字符量计费 | TTS | `unit_price × Math.ceil(char_count / billing_unit_chars)` | 单价、计费单位（如1000字符） |
| **T4** | 按数量计费 | 图像生成（每个规格一行） | `unit_price × metric_volume` | 单价 |
| **T5** | 阶梯计费 | OCR大量、ASR大量 | 按月累计量分段取单价 × metric_volume | 阶梯规则、阶梯模式 |
| **T6** | 自定义公式 | 视频生成(TODO)、复杂场景 | QLExpress表达式 | 公式文本、变量绑定 |

### 3.3 模板说明

#### T1 按次计费

```
points = unit_price × metric_volume
```

- metric_volume：本次调用的数量（如调用1次=1，调用3次=3）
- 最简单的场景，配一个单价即可

#### T2 按时长计费

```
points = unit_price × Math.ceil(duration_seconds / billing_unit_seconds)
```

- duration_seconds：本次调用的实际时长（秒），从计量事件中取
- billing_unit_seconds：计费单位（秒），配置项。如讯飞ASR=15秒、Google STT=15秒
- Math.ceil：向上取整（不足一个计费单位按一个计费单位算）
- 举例：实际通话38秒，billing_unit=15 → ceil(38/15)=3个计费单位 → unit_price×3

#### T3 按字符量计费

```
points = unit_price × Math.ceil(char_count / billing_unit_chars)
```

- char_count：本次调用处理的字符数，从计量事件中取
- billing_unit_chars：计费单位（如1000字符），配置项
- 举例：处理2500个字符，billing_unit=1000 → ceil(2500/1000)=3 → unit_price×3

#### T4 按数量计费

```
points = unit_price × metric_volume
```

- 和T1公式相同，但**配置语境不同**
- T1用于"调用次数"，T4用于"生成数量"（如图片张数）
- 图像生成每个分辨率独立配置一行，通过"服务标识+输出规格"区分

#### T5 阶梯计费

按月累计用量分段，取对应段的单价：

```
points = tier_price(current_monthly_total) × metric_volume
```

**两种阶梯类型**：

| 类型 | 名称 | 规则 | 典型场景 |
|------|------|------|---------|
| **周期内阶梯** | 周期清零 | 固定周期内累计量决定单价，周期结束清零重新计 | 厂商按月/季度计费（如你截图的月调用量分段） |
| **累计阶梯** | 永不清零 | 历史总调用量决定单价，永不清零 | 会员等级式（用得越多越便宜，不随月份重置） |

**周期内阶梯的周期选项**：

| 周期 | 说明 |
|------|------|
| 自然月 | 每月1日0:00清零 |
| 自然周 | 每周一0:00清零 |
| 自然季度 | 每季度首日0:00清零 |

**两种阶梯模式（定价结构）**：

| 模式 | 名称 | 规则 | 说明 |
|------|------|------|------|
| **超额计费** | 电费模式 | 每次按"当前累计量所在段"的单价计费 | 用多少算多少 |
| **全量计费** | 档位模式 | 当前调用按"当前累计量所在段"的单价计费 | 同上（实时扣点场景下行为一致） |

**超额计费示例**：
- 阶梯：0~200免费，201~10000按0.13，10001+按0.10
- 本月累计第5000次 → 当次落入"201~10000"段 → 本次0.13点
- 第10001次 → 进入新段 → 本次0.10点
- 同一个月内不同次可能按不同段价计费

**全量计费示例**：
- 阶梯：月用量0~100按1.0/次，101~1000按0.8/次，1001+按0.5/次
- 本月到了第500次 → 当前在"101~1000"段 → 所有后续调用都按0.8
- 不会回头重算前面的

**阶梯累计规则**：
- 累计粒度：**按租户**（厂商对我们收费是租户级别）
- 累计周期：**按配置**（周期内阶梯：自然月/周/季度，到期清零；累计阶梯：永不清零）
- 累计范围：同一"计费类型 + 供应商 + 服务标识 + 输出规格"的所有调用

#### T6 自定义公式

```
points = [用户编写的QLExpress表达式]
```

- 配置人员编写完整公式表达式
- 变量从计量事件的event_data中自动注入
- 用于覆盖未来复杂场景（如视频生成：时长×分辨率×帧率等）
- 需要支持**预览计算**功能：输入模拟数据，实时查看结果，验证公式正确性

## 四、配置维度

### 4.1 完整维度清单

| # | 维度 | 含义 | 举例 | 为空时的含义 |
|---|------|------|------|-------------|
| 1 | **计费类型** | 哪种服务 | WEBSEARCH / OCR / IMAGE_GEN / ASR / TTS / WPS | 必填 |
| 2 | **供应商** | 谁提供的服务 | 百度 / 讯飞 / 即梦 / Google | 必填 |
| 3 | **服务标识** | 厂商侧的API/产品标识 | seedance-2.0 / baidu_ocr / web_search | 必填 |
| 4 | **输出规格** | 同一服务下不同定价的档位 | 720p / 1080p / 通用 / 发票 | 无规格差异时为空 |
| 5 | **模板类型** | 使用哪个模板 | T1~T6 | 必填 |
| 5 | **单价（unit_price）** | 每计费单位的点数 | 0.67 | T5/T6时可空 |
| 6 | **计费单位值** | 取整时的单位量 | 15（秒）/ 1000（字符） | T2/T3时必填 |
| 7 | **计费单位名称** | 展示用 | 次 / 15秒 / 千字符 / 张 | 选填 |
| 8 | **阶梯规则** | 分段的阈值+单价列表 | 见下方格式 | T5时必填 |
| 9 | **阶梯类型** | 周期内阶梯 or 累计阶梯 | PERIODIC / CUMULATIVE | T5时必填 |
| 10 | **阶梯周期** | 周期内阶梯的清零周期 | 自然月 / 自然周 / 自然季度 | 仅周期内阶梯必填 |
| 11 | **阶梯模式** | 超额计费 or 全量计费 | INCREMENTAL / TIERED | T5时必填 |
| 10 | **公式文本** | 自定义表达式 | `base * ceil(duration/10) * factor` | T6时必填 |
| 11 | **公式版本** | 系统自动生成，追溯用 | v1.0.20260703 | 自动 |
| 12 | **状态** | 启用/停用 | ENABLED / DISABLED | 默认启用 |

### 4.2 阶梯规则格式

```json
{
  "mode": "INCREMENTAL",
  "tiers": [
    { "threshold": 200, "unit_price": 0 },
    { "threshold": 10000, "unit_price": 0.13 },
    { "threshold": -1, "unit_price": 0.10 }
  ]
}
```

- threshold=-1 表示"无上限"（最后一段）
- unit_price 为该段的每次点数
- mode: INCREMENTAL（超额计费）/ TIERED（全量计费）

### 4.3 唯一标识

```
计费类型 + 供应商 + 服务标识 + 输出规格
```

### Case 5：语音识别（按时长取整 + 阶梯）

**场景**：讯飞ASR，按15秒为计费单位，月累计阶梯

**配置**：

| 计费类型 | 供应商 | 服务标识 | 模板 | 计费单位值 | 阶梯模式 | 阶梯规则 |
|---------|--------|---------|------|-----------|---------|---------|
| ASR | 讯飞 | 标准版 | T5 | 15（秒） | INCREMENTAL | [{3600, 0.1}, {36000, 0.08}, {-1, 0.06}] |

> 注：阶梯的threshold单位是"计费单位数"（即15秒为1个计费单位）。3600个计费单位=15小时。

```
本月已用2000个计费单位，本次通话38秒：
  取整：ceil(38/15) = 3 个计费单位
  月累计：2000+3=2003 → 落入"0~3600"段 → 单价0.1
  → points = 0.1 × 3 = 0.3点 ✅
```

**关键**：
- 计量事件传**原始秒数**（duration_seconds=38）
- 系统先做取整（ceil(38/15)=3）得到本次的metric_volume
- 再用metric_volume去查阶梯和计算

## 六、阶梯累计量的业务规则

### 6.1 累计维度

| 维度 | 值 | 说明 |
|------|-----|------|
| 累计粒度 | **租户** | 同一租户下所有用户的调用合并累计 |
| 累计周期 | **按配置** | 周期内阶梯：自然月/周/季度（到期清零）；累计阶梯：永不清零 |
| 累计范围 | **同一（计费类型+供应商+服务标识+输出规格）** | 不同配置行分别累计 |

### 6.2 两种阶梯模式对比

| | 超额计费(INCREMENTAL) | 全量计费(TIERED) |
|---|---|---|
| 规则 | 每次按"当前量所在段"的单价计费 | 每次按"当前量所在段"的单价计费 |
| 区别 | 跨段时，前面段已按前段价算过，不追溯 | 同上（实际行为相同） |
| 适用 | 调用量大、单价递减的场景（鼓励多用） | 同上 |

> 说明：对于"每次单独计费"的场景（每次调用独立计算），超额计费和全量计费的**当次计算结果实际相同**——都是取当前累计量所在段的单价。区别仅在"批量一次性结算"场景（如月底一次性结算当月全部调用），此时超额计费要分段计算、全量计费按最终段统一价。
> 
> 我们的场景是**每次调用实时计费**，所以两种模式在实时扣点时行为一致。保留两种模式是为了：
> 1. 配置时语义明确
> 2. 未来支持批量结算/对账场景时有区分能力

## 七、与已有系统的关系

### 7.1 数据流

```
业务方 → Console计量接口（委托授权，带鉴权）
           ↓
       转发到外部服务执行
           ↓
       记录计量事件到 t_meter_tenant_user_other_metering
           ↓
       查 t_meter_sys_metering_formula 匹配公式配置
           ↓
       按模板/公式计算点数
           ↓
       写流水表 + 扣点 + 刷新Redis
```

### 7.2 表关系

| 表 | 作用 |
|----|------|
| t_meter_sys_metering_formula | 公式配置（本方案核心） |
| t_meter_tenant_user_other_metering | 非LLM计量记录 |
| t_meter_tenant_user_statement | 流水表（复用，metering_type区分） |

### 7.3 计量事件需要携带的信息

| 字段 | 说明 | 来源 |
|------|------|------|
| metering_type | 计费类型 | Console计量接口根据调用路由确定 |
| provider | 供应商 | Console根据接入配置确定 |
| service_spec | 服务标识 | Console根据请求参数确定（如分辨率→2048） |
| metric_volume | 数量 | 外部服务返回（如生成4张=4） |
| event_data | 原始计量数据（秒数/字符数等） | 外部服务返回 |
| event_id | 幂等ID | 生成或外部返回 |

## 十一、管理页面交互设计

详细原型见 `产品侧/metering-formula-prototype.html`。

### 核心功能

| 功能 | 说明 |
|------|------|
| 公式列表 | 按计费类型一级分组，显示供应商·规格 + 模板标签 + 单价 |
| 创建公式 | 选类型+供应商+规格 → 选模板 → 填参数 → **即时试算验证** → 创建 |
| 编辑公式 | 修改参数，含生效方式选择（立即/定时） |
| 即时试算 | 创建和编辑时均可输入模拟数据实时计算，验证公式正确性 |
| 启用/禁用 | 每条配置支持开关切换 |
| 版本历史 | 每次变更保存新版本，可点击查看历史快照 |
| 导入导出 | CSV/JSON批量管理 |
